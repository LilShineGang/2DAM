Arquitectura Cliente/Servidor
-----------------------

Los clientes hacen peticiones/request en el protocolo http al servidor web. Lo que nos devuelve el servidor son respuestas/response.

Características
-----------------------

- Protocolo orientado a transacciones: esquema petición <-> respuesta
- Petición = Request
- Respuesta = Response
- Cliente (user agent) y servidor
- Información que se transmite = recurso
- El recurso se identifica por medio de una URL.
- Protocolo sin estado: no guarda conexiones anteriores, lo que lo hace escalable aunque, al mismo tiempo, es su principal limitación que mitigaremos por medio del desarrollo web (cookies)

DESGRANANDO UN ENDPOINT

- https:/api.library.com/v1/books/10 format=summary&available=true

- http -> Protocolo
- api.library.com -> Dominio
- /v1/books/10 -> Recurso solicitado
- ?format=summary&available=true -> Query parameters

El valor 10 del recurso /v1/books/10 es lo que se denomina PATH PARAMETER y, en este caso, es evidente que es el identificador interno del libro.

UTILIZA RECURSOS RESTFUL

Nombres de recursos: sustantivos en plural
- /usuarios
- /productos

Métodos HTTP adecuados: GET, POST, PUT (o PATH), DELETE

MENSAJE DE ERRORES:

Códigos de estado HTTP:
- 200 -> éxito
- 201 -> creación exitosa
- 400 -> solicitud incorrecta
- 404 -> recurso no encontrado
- 500 -> error del servidor

Mensajes de error: Proporciona mensajes de error descriptivos en el cuerpo de la respuesta.
